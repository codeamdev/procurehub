prompt 2:

Enhance the existing multi-tenant Django + Next.js application by implementing a robust permission system based on roles and workflow steps.

## Goal

Implement fine-grained permissions that control:

* Who can view a request
* Who can edit data in a step
* Who can execute actions in a step

---

## User Roles

Use these roles:

* admin
* buyer
* supplier

Ensure role is stored in the User model.

---

## Backend تغييرات (Django)

### Extend WorkflowStep

Add permission fields:

* allowed_roles_to_view (JSONField or ArrayField)
* allowed_roles_to_edit (JSONField or ArrayField)
* allowed_roles_to_act (JSONField or ArrayField)

Example:

{
"allowed_roles_to_view": ["admin", "buyer"],
"allowed_roles_to_edit": ["buyer"],
"allowed_roles_to_act": ["buyer"]
}

---

## Request Access Rules

### Create utility functions:

* can_view_request(user, request)
* can_edit_step(user, request)
* can_execute_action(user, request)

These should:

* Check user.role
* Check current step permissions
* Return True/False

---

## Protect API Endpoints

Apply permission checks in:

### 1. Get Request Detail

* Only allow if can_view_request == True

### 2. Update Request Data

* Only allow if can_edit_step == True

### 3. Execute Action Endpoint

* Only allow if can_execute_action == True

If not allowed:

* Return HTTP 403 with clear message

---

## Supplier Restrictions

* Suppliers can ONLY:

  * View requests assigned to them
  * Submit responses
* Suppliers CANNOT:

  * Approve/reject workflows
  * Modify buyer data

---

## WorkflowAction Security

Before executing an action:

* Validate that user role is allowed
* Validate that action belongs to current step

---

## Frontend (Next.js)

### Hide UI based on permissions

* If user cannot edit:

  * Disable form inputs

* If user cannot act:

  * Hide action buttons

* If user cannot view:

  * Redirect or show "Access Denied"

---

## API Response Enhancement

When returning request detail, include:

{
...
"permissions": {
"can_view": true,
"can_edit": false,
"can_act": true
}
}

---

## UX Improvements

* Disable buttons instead of removing (optional)
* Show tooltip: "You don't have permission"

---

## Important Rules

* NEVER trust frontend alone
* Always enforce permissions in backend
* Keep logic reusable and centralized

---

## Clean Code

* Use service layer or utils for permission logic
* Avoid duplicating permission checks
* Add comments explaining permission flow

---

## Output

* Secure backend endpoints
* Permission-aware frontend UI
* Clean, scalable permission system ready for production

- Multi-tenant architecture using django-tenants (schema per tenant)

- Frontend: React (Vite)
- Python 3.12 virtual environment (no Docker)

🔐 BACKEND REQUIREMENTS (Django)
-----------------------------------

1. Authentication System:

  - Login endpoint
  - Refresh token endpoint
  - Logout (blacklist tokens)

- Extend AbstractUser
- Fields:
  - email (unique, used as username)
  - role (choices below)
  - company_name (optional for providers)
- Roles:
  - ADMIN (buyer company admin)
  - BUYER (creates requests)
  - SUPPLIER (submits proposals)

3. Multi-tenant Awareness:
- Users belong to a tenant (schema)
- Ensure queries are tenant-isolated

      - tenant (schema / company)

- Create custom DRF permissions:
  - IsAdmin
  - IsBuyer
  - IsSupplier
- Enforce:
  - Only BUYER/ADMIN can create requests
  - Only SUPPLIER can submit proposals
  - Only ADMIN can approve suppliers

- Supplier signs up
- Requests access to a tenant
- Admin approves/rejects
- Only approved suppliers can see and bid on requests

6. API Endpoints:
- POST /auth/register/
- POST /auth/login/
- GET /auth/me/
- POST /suppliers/request-access/

- POST /suppliers/{id}/reject/

🧠 FRONTEND REQUIREMENTS (React)
-----------------------------------

1. Auth Pages:
- Login page
- Register page (supplier + buyer options)

2. Auth Context:
- Store JWT tokens
- Auto-refresh tokens
- Persist session

3. Role-based UI:
- ADMIN:
  - Manage suppliers
- BUYER:
  - Create requests
- SUPPLIER:
  - View available requests
  - Submit proposals

4. Route Protection:
- Protected routes based on role
- Redirect if unauthorized

-----------------------------------
🔐 SECURITY BEST PRACTICES
-----------------------------------

- Hash passwords properly (default Django)
- Use HTTP-only cookies OR secure token storage
- Validate all inputs
- Prevent cross-tenant data leaks
- Add basic rate limiting (optional but recommended)

-----------------------------------
📁 OUTPUT FORMAT
-----------------------------------

1. Django:
   - Updated User model
   - Models for SupplierCompanyAccess
   - Serializers
   - Views (DRF ViewSets or APIViews)
   - Permissions classes
   - URLs

   - AuthContext
   - Login/Register pages

   - Example role-based dashboard rendering

3. Setup Instructions:
   - Install dependencies
   - Run migrations
   - Create superuser

Keep code clean, modular, and production-ready.
